## Manual D365 sandbox step (unchanged from previous iteration, re-listed for clarity)

After rebuilding `MultiYearDeal`, update the assembly in the Plugin Registration
Tool. The two Custom APIs (`ats_CheckProductsAvailability`, `ats_AddProductsBatch`)
stay — **but `ats_AddProductsBatch` now accepts an optional third input
parameter**: `softTimeoutMs` (Edm.Int32, optional). Add it to the Custom API
definition if you want the PCF to be able to override the default 90 s
server-side soft-timeout. The PCF doesn't require it — the default is fine
for typical batches.

## Backward compatibility

- The existing response fields the PCF reads (`success`, `agreementId`,
  `createdOpportunityProductIds`, `touchedOpportunityIds`, `message`,
  `errorCode`) are all preserved — they just shifted casing from mixed to
  lowercase and are now strictly consistent with `CheckProductsAvailability`.
- `skippedPackageProductIds` was removed from the response because the backend
  no longer skips anything — if the PCF previously inspected this field for
  anything, the new flow's `failedProducts[]` + the fact that all packages
  land in `createdOpportunityProductIds` covers the same signal.
- The legacy `ats_customAPIAddProductAgreementCartAction` endpoint is still
  registered and untouched — if you need to roll back, the only PCF file to
  revert is `AgreementTable/index.tsx` (the new flow can be replaced with
  the prior per-product loop and `executeAddProductFlow` stays exported).

## Recommended verification in the sandbox

1. **Auto-gen sanity** — Stage a product that has NO IBS in the target season
   but DOES have an IBS in another season. Submit. Expect: the pre-flight
   auto-creates the IBS (check for `ats_autogenerated = true` on the new row),
   and the submission succeeds.
2. **No-template guard** — Stage a product that has NO IBS anywhere. Expect:
   pre-flight returns `ibs_missing` and the PCF aborts with a clear message.
3. **Package submission** — Stage a package product with 3+ components.
   Submit. Expect: 1 main `opportunityproduct` + N component rows per
   opportunity, all linked via `ats_packagelineitem` to the main row; all
   tagged with the same `ats_agreementopportunityproduct` value.
4. **Soft-timeout loop** — Stage ~100 products and submit. In the Network tab
   expect multiple sequential POSTs to `ats_AddProductsBatch`, each with a
   shrinking `products` array. The progress bar should update after each call.
   If you want to test the yield explicitly, lower `softTimeoutMs` to e.g.
   `5000` in the `executeAddProductsBatchUntilComplete` options to force
   many small sub-calls.
5. **Partial failure** — Corrupt one of the products' `RateId` (or send an
   invalid GUID) so the transaction fails for that one. Expect: `failedProducts`
   contains that product, the others succeed, and the PCF keeps the failed one
   in the staging area for retry.
